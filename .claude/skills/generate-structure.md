# Generate Game Design Document Structure

สร้างโครงสร้าง folder และไฟล์ template สำหรับ game design document โดยอ่านข้อมูลจาก `design.md`

## เมื่อไหร่ควรใช้

ใช้เมื่อ user ต้องการ scaffold โครงสร้าง game design document, สร้างไฟล์ห้องใหม่, หรือสร้าง map file จากข้อมูลใน `design.md`

## ขั้นตอน

### 1. อ่าน design.md

อ่านไฟล์ `design.md` จาก root ของ repo หากไม่พบ ให้ถาม user ว่าไฟล์ design overview อยู่ที่ไหน

### 2. Extract ข้อมูลจาก design.md

ดึงข้อมูลต่อไปนี้:

- **ข้อมูลเกม:** ชื่อเกม, แนว (genre), มุมมอง (perspective), แพลตฟอร์ม
- **รายชื่อห้อง:** จาก code block ใน section "โครงสร้างไฟล์" — แต่ละห้องมีเลข prefix และชื่อ snake_case (เช่น `01_bedroom.md`)
- **ลำดับการปลดล็อค:** จาก flow diagram หรือ section ลำดับการปลดล็อค
- **ปริศนาหลัก:** ปริศนาที่ร้อยเรื่องข้ามห้อง (เช่น รหัส 4 หลักที่ต้องรวบรวมจากหลายห้อง)
- **กลไกหลัก:** ระบบ inventory, สมุดบันทึก, การตาย, บรรยากาศ
- **การเชื่อมต่อห้อง:** ห้องไหนเชื่อมกับห้องไหน พร้อมเงื่อนไข

### 3. สร้าง Directory Structure

- สร้าง `rooms/` directory หากยังไม่มี
- สร้าง directory อื่นๆ ที่ระบุใน design.md (ถ้ามี)

### 4. Generate map.md

สร้างไฟล์ `map.md` ที่ root โดยมีโครงสร้าง:

```markdown
# <ชื่อเกม> — แผนผังห้องและการเชื่อมต่อ

## แผนผังหลัก (Mermaid)

(Mermaid graph TD diagram แสดงห้องทั้งหมดและการเชื่อมต่อ
- แต่ละ node ใช้ emoji + ชื่อไทย + ชื่ออังกฤษ
- edges แสดงเงื่อนไขการเข้า)

## ตารางการเชื่อมต่อ

| ห้อง | เชื่อมต่อกับ | เงื่อนไขการเข้า |
|------|------------|----------------|

## แผนผัง Backtracking (การย้อนกลับ)

(Mermaid graph LR diagram แสดง cross-room dependencies)

## สถานะห้อง (Room States)

| สัญลักษณ์ | ความหมาย |
|-----------|----------|
| 🔒 | ยังเข้าไม่ได้ |
| 🔓 | ปลดล็อคแล้ว เข้าได้ |
| ✅ | ทำปริศนาหลักครบแล้ว |
| 🔄 | มีไอเทม/เบาะแสใหม่ปรากฏ (เมื่อกลับมา) |
| ⚠️ | อันตราย — ต้องระวัง |

## ลำดับเหตุการณ์โดยรวม (High-Level Flow)

(ASCII flow diagram แสดงลำดับการปลดล็อคห้องตั้งแต่ต้นจนจบ)
```

### 5. Generate Room Files

สร้างไฟล์แต่ละห้องใน `rooms/` ตาม template ด้านล่าง

#### Template ห้องมาตรฐาน

```markdown
# <ชื่อห้องไทย> (<English Room Name>)

## บรรยากาศ
<!-- อธิบายบรรยากาศของห้อง: แสง สี เสียง กลิ่น ความรู้สึก -->

## สถานะเริ่มต้น
<!-- สำหรับห้องแรก ใช้ "สถานะเริ่มต้น" / สำหรับห้องอื่น ใช้หัวข้อ "เงื่อนไขการเข้า" แทน -->
- (ระบุเงื่อนไขหรือสถานะเริ่มต้นของห้อง)

## วัตถุที่โต้ตอบได้

| วัตถุ | การกระทำ | ผล |
|------|----------|-----|
| **ตัวอย่าง** | ตรวจสอบ | ผลลัพธ์ |

## ปริศนา

### ปริศนา A — (ชื่อปริศนา)
- **เงื่อนไข:** ...
- **วิธีแก้:** ...
- **รางวัล:** ...

## เงื่อนไขการตาย

| การกระทำ | สาเหตุการตาย |
|----------|-------------|
| ตัวอย่าง | สาเหตุ |

## สิ่งที่เปลี่ยนเมื่อกลับมา

- ครั้งที่ 2: ...
- ครั้งที่ 3+: ...

## Backtracking จากห้องอื่น

| กลับมาจาก | เหตุผล | ไอเทมที่ต้องการ |
|-----------|--------|----------------|
| ห้อง... | เหตุผล... | ไอเทม... |

## ไอเทมที่ได้จากห้องนี้

- `ชื่อไอเทม` → ใช้ที่ไหน / อย่างไร
```

#### การปรับ template ตามบทบาทของห้อง

- **ห้องแรก (starting room):** ใช้ "## สถานะเริ่มต้น" แทน "## เงื่อนไขการเข้า"
- **ห้องสุดท้าย (ending room):** เพิ่ม section:
  - `## จบเกม (Endings)` — อธิบาย ending ต่างๆ
  - `## หมายเหตุการออกแบบ` — โน้ตสำหรับทีมออกแบบ
- **ห้องที่มีปริศนาสำคัญ (major puzzle room):** เพิ่ม `## หมายเหตุการออกแบบ`

### 6. Pre-fill ข้อมูลจาก design.md

ถ้า design.md มีข้อมูลเพียงพอ ให้ใส่ลงใน template เลย:

- **เงื่อนไขการเข้า:** ดึงจากลำดับการปลดล็อค
- **ปริศนาหลัก:** ถ้า design.md ระบุว่าห้องนี้มีตัวเลขหลักที่ N ของรหัส ให้ใส่ใน Interactive Objects
- **การเชื่อมต่อ:** ดึงจาก flow diagram เพื่อระบุว่าห้องนี้เชื่อมกับห้องไหนบ้าง
- **ไอเทม:** ถ้า design.md กล่าวถึงไอเทมจากห้องนี้ ให้ใส่ใน section ไอเทม

### 7. กฎสำคัญ

- **ห้ามเขียนทับไฟล์ที่มีอยู่แล้ว** — ถ้าไฟล์มีอยู่แล้ว ให้ skip และแจ้ง user ว่า "ข้ามไฟล์ X เพราะมีอยู่แล้ว"
- **ใช้ภาษาเดียวกับ design.md** — ถ้า design.md เป็นภาษาไทย ให้ generate เป็นภาษาไทย
- **Filename pattern:** ถ้า design.md มี code block ระบุชื่อไฟล์ ให้ใช้ตามนั้น ถ้าไม่มี ให้ใช้ pattern `NN_snake_case_name.md` โดย NN เป็นเลขลำดับ zero-padded
- **ทำงานได้กับเกมอะไรก็ได้** — ไม่ hardcode เฉพาะเกมใดเกมหนึ่ง ให้ดึงข้อมูลจาก design.md เสมอ
